# Task Principles

A task is a focused phase of work under a hardened milestone, sized for one human-agent collaboration cycle when possible, that meaningfully advances the milestone.

## Principles

### Advance the milestone

A task should clearly move its milestone forward. If it does not connect to the milestone's value or done condition, it probably belongs elsewhere.

### Phase, not micro-todo

A task can contain many small steps. It should represent a meaningful chunk of progress, not a tiny implementation checklist.

Good:

> Prototype correction review flow.

Too small:

> Add button.

### Clear outcome

A task should state what should be true when it is done. The outcome can be an artifact, decision, working implementation, completed experiment, recommendation, or clarified issue.

### Discovery is allowed

A task does not need a fully known path. Exploratory tasks are valid when they have a clear learning or decision outcome.

### Directional by default

A task should usually describe the outcome and direction, not every implementation step. Include known constraints or suggested approaches, but leave room for the human-agent session to discover the best path.

### Incremental value slices

When shaping task lists or phase-by-phase plans, prefer tasks that deliver, validate, or de-risk a small usable slice of the milestone end-to-end. Avoid horizontal layer-by-layer plans as the default. Use horizontal tasks only when they are necessary to unlock or de-risk later value.

### Collaboration-sized

Aim for one focused human-agent session. Spillover is allowed; if a task obviously spans many sessions, split it.

### Human, agent, or mixed

Tasks may be agent-heavy, human-heavy, or mixed. Do not encode actor metadata unless the project has its own convention.

### Lightweight dependencies

Capture dependencies only when they affect sequencing. Avoid dependency graphs unless they reduce confusion.

### Self-contained when persisted

A persisted task should be self-contained enough that future-you or a future agent can reorient from the task file alone. It may contain uncertainty, but it should not be so vague that the original intent is lost.

## Quality Checks

A useful task can answer:

1. What milestone does this advance?
2. What phase of progress does this represent?
3. What should be true after this task?
4. Can this fit in one focused human-agent cycle?
5. What, if anything, must happen before it?

## Pushback

If the milestone seems clear but useful tasks are hard to define, the milestone may not be hardened yet. Revisit the milestone's boundaries, done condition, or rough direction instead of inventing tasks prematurely.
