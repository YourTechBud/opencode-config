# Milestone Principles

A milestone is a lightweight continuation marker for a valuable direction of work. It should help future-you resume without re-discovering why the work matters.

## Principles

### Value is the center

A milestone exists because something becomes better, clearer, easier, validated, shipped, fixed, taught, or unlocked. If the value is unclear, stay in discovery.

### Direction-shaped, not task-shaped

A milestone is not just a list of tasks. It names a valuable direction and the rough path toward it. Tasks serve the milestone; they are not the milestone.

### Directional by default

A milestone should preserve intent, value, boundaries, and continuation without freezing the implementation path too early. Add implementation detail only when the user has chosen it, it is already known, or precision reduces risk.

### Coherent center of gravity

A milestone should hold together around one value direction. If it is a bucket of unrelated work, treat it as a release, backlog, category, or cleanup list instead.

### Done condition

A milestone needs a signal for when to stop, move on, kill, or reshape. Done can mean shipped, fixed, learned, validated, rejected, decided, made usable enough, or clarified enough to reshape.

### Boundaries prevent sprawl

A useful milestone says what belongs now vs later. Park valuable-but-premature ideas as future candidates instead of expanding the current milestone until it becomes a blob.

### Continuation over completeness

A milestone does not need a perfect plan. It needs enough context that future-you can tell why it matters, what direction it points, and where to continue.

### Reduce mental load

Milestones exist to preserve momentum. If defining or maintaining one feels heavier than the clarity it creates, simplify it.

## Quality Checks

A useful milestone can answer, briefly:

1. Why this?
2. What direction?
3. What belongs now vs later?
4. Where do I continue?
5. How do I know this is done enough?
6. Does this reduce mental load?
